# 01 — Исходные данные (Product Brief)

> «Что и зачем» — до того, как думать «как». Статус документа: 🟡 черновик

## 1. Проблема

Частная стоматологическая клиника. Главная боль — **запись на приём и видимость записей**:
- Запись идёт только по звонку администратору.
- Администратор вносит пациента в **MIS** — коммерческую медицинскую информационную систему (доступа к ней у пациента нет).
- Пациенту сообщают дату устно, он записывает её «на бумажке».
- Нет удобного способа увидеть свои текущие/прошедшие/предстоящие записи и получать напоминания.

Расписание врачей не структурировано в удобном для клиники и пациента виде.

## 2. Целевая аудитория

B2C, частная стоматология (одна клиника на старте).

| Сегмент | Кто это | Главная потребность |
|---------|---------|---------------------|
| Пациент | Клиент клиники | Видеть свои записи (прошедшие/текущие/будущие), получать напоминания |
| Врач | Сотрудник клиники | Видеть своё структурированное расписание и записи пациентов |
| Администратор клиники | Связующее звено | Управлять расписанием и записями (вносит запись в MIS) |
| Администратор сервиса | Технический специалист | Поддерживать работу приложения |

## 3. Ценностное предложение

Удобный интерфейс поверх MIS: единая точка доступа к расписанию и записям для **врачей, администратора и пациентов**. Человеческий UX над системой, к которой у пациентов сейчас нет прямого доступа.

## 4. Цели и задачи

- Структурировать расписание врачей.
- Дать пациентам видимость своих записей и напоминания.
- Снять с администратора рутину «звонок → запись в MIS → продиктовать дату».

## 5. Scope

### В рамках (MVP)
- Просмотр календаря записей пациентов к врачам.
- Просмотр истории записей (прошедшие/текущие/предстоящие).
- Оповещения о записях.
- Запись на приём выполняет **администратор** (пациент в MVP только просматривает).
- Роли: пациент, врач, администратор клиники, администратор сервиса.

### Вне рамок (осознанно — позже)
- Самостоятельная запись пациента через приложение.
- Хранение медицинских документов / доступ к ним через внешние мед. системы.
- Чат с врачом, видеоконференцсвязь.
- Мобильное нативное приложение (на старте — адаптивный веб).
- Использование «продажной» части API MIS (нужно только расписание).

## 6. Метрики успеха (MVP)

- **Главная:** несколько десятков активных пациентов, реально пользующихся приложением.
- **Качественная:** положительные отзывы этих пациентов.
- Вспомогательные (косвенно подтверждают ценность): доля валидированных пациентов (T2), доля заходящих посмотреть свои записи, снижение обращений к администратору «когда у меня приём».

## 7. Ограничения и допущения

- Интеграция с **MIS** — коммерческой МИС, используемой многими клиниками и интегрированной с государственным учётом приёмов в частных клиниках.
- Из API MIS в MVP нужен **только модуль расписания** (продажи и пр. — вне scope).
- Наше приложение — интерфейс доступа, а не замена MIS.
- Пользователи в **России** → 152-ФЗ; медданные — особая категория ПДн, хранение на территории РФ.

- ❓ _Бюджет, сроки, команда — уточняются._
