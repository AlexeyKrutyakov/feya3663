# 04 — Поэтапный план (Roadmap)

> Фазы → задачи. Каждая задача прослеживается до требования из `02-requirements.md`. Статус: 🟡 черновик

## Принципы планирования

- Каждая фаза заканчивается **работающим, проверяемым результатом**.
- MVP = минимум, решающий проблему пользователя (см. `01`).
- Интеграция с MIS — на критическом пути; её риски снимаем как можно раньше.
- Сначала «вертикальный срез» (чтение записей end-to-end), потом расширение.

## Обзор фаз

| Фаза | Цель | Ключевой результат | Статус |
|------|------|--------------------|--------|
| 0. Фундамент | Каркас монорепо, окружение, CI | Пустые fe/be деплоятся, общие типы | ⬜ |
| 1. Интеграция MIS (чтение) | Снять главный риск | Записи из MIS видны в БД | ⬜ |
| 2. Доступ и роли | Аутентификация + тиры | T0/T1/T2 + RBAC работают | ⬜ |
| 3. Кабинеты (чтение) | Ценность для пользователя | Пациент/врач видят свои записи | ⬜ |
| 4. Уведомления | Закрыть боль «забыл приём» | SMS + Telegram о записях | ⬜ |
| 5. Управление записями (write-back) | Админ работает только у нас | Создание/перенос/отмена → MIS | ⬜ |
| 6. Лендинг + 3D | Маркетинг и воронка T0→T1 | Публичная страница с 3D-феей | ⬜ |
| 7. Стабилизация | Готовность к проду | Соответствие 152-ФЗ, наблюдаемость | ⬜ |

> Порядок фаз 1→2 можно вести параллельно (разные подсистемы). Лендинг (6) можно начинать рано визуально, но он зависит от воронки доступа (2).

## Фаза 0 — Фундамент
_Связано с: ADR-001._
- [ ] Монорепо: pnpm workspaces + Turborepo; пакеты `web`, `api`, `shared`.
- [ ] `api`: каркас NestJS, Prisma + PostgreSQL (docker-compose для локали), Redis.
- [ ] `web`: каркас Next.js + Tailwind, базовый layout.
- [ ] `shared`: первые Zod-схемы и общие типы.
- [ ] Линт/формат/тайпчек, Vitest, заготовка Playwright.
- [ ] CI (GitHub Actions): сборка, тесты, тайпчек. Коммиты — `/commit`.
- **DoD:** оба приложения поднимаются локально и в CI; общий пакет импортируется.

## Фаза 1 — Интеграция MIS: чтение
_Связано с: FR-14; ADR-002; `06-mis-api-reference.md`._
- [ ] MIS-клиент: авторизация (`/auth`), хранение токена клиники (безопасно).
- [ ] Импорт `resource`/`service`/`visit` за период → проекция (Prisma-модели).
- [ ] Приём вебхуков `/webhooks/mis`: дедуп (`WebhookEvent`), идемпотентность по `misVisitId`+`update_date`.
- [ ] Периодическая сверка (BullMQ-задача) на случай пропущенных вебхуков.
- [ ] Нормализация часовых поясов `datetime`.
- **DoD:** запись, созданная в MIS, появляется/обновляется в нашей БД (вебхук и/или сверка). Зависит от Q5a (способ доступа).

## Фаза 2 — Доступ и роли
_Связано с: FR-1, FR-2, FR-3, FR-5; ADR-003._
- [ ] Регистрация email+пароль, подтверждение email, сессии (httpOnly cookie), сброс пароля.
- [ ] Модель `User`/`PatientProfile`, тиры T0/T1/T2, RBAC-гард в NestJS.
- [ ] Анкета пациента (T1) + указание телефона.
- [ ] Очередь валидации у админа (approve/reject/очно) → переход T1→T2.
- [ ] Сопоставление с MIS-клиентом по телефону при валидации.
- **DoD:** гость регистрируется, заполняет анкету, админ валидирует, доступ открывается.

## Фаза 3 — Кабинеты (чтение)
_Связано с: FR-6, FR-7, FR-9, FR-10._
- [ ] Кабинет пациента: предстоящие/прошедшие записи (дата, время, врач, услуга).
- [ ] Кабинет врача: расписание день/неделя по своему `resourceId`, карточка записи.
- [ ] Адаптивная вёрстка.
- **DoD:** валидированный пациент видит свои записи; врач — своё расписание.

## Фаза 4 — Уведомления (SMS)
_Связано с: FR-8, FR-15._
- [ ] Сервис уведомлений: модель `Notification`, планирование (BullMQ).
- [ ] SMS-канал (рос. провайдер — Q8a) с абстракцией провайдера.
- [ ] Триггеры: напоминание заранее, уведомление об изменении/отмене (из вебхуков).
- **DoD:** при появлении/изменении записи пациент получает SMS.
- _Telegram (FR-15b, ADR-004) — после MVP: отдельный VPS-прокси._

## Фаза 5 — Управление записями (write-back)
_Связано с: FR-11, FR-12, FR-13; ADR-002._
- [ ] Админ-UI: создание/перенос/отмена записи; выбор свободных слотов (`/resource/{id}/date`,`/time`).
- [ ] Запись в MIS (`POST/PUT/DELETE /visit`) через очередь с повторами.
- [ ] Согласование состояния с подтверждающим вебхуком; обработка ошибок/гонок.
- **DoD:** админ полностью ведёт записи в нашем приложении, изменения уходят в MIS.

## Фаза 6 — Лендинг + 3D
_Связано с: тир T0, FR-1 (воронка)._
- [ ] Маркетинговый лендинг (SSR/SEO, Next.js).
- [ ] 3D-фея на React Three Fiber (оптимизация загрузки, fallback).
- [ ] Призывы к регистрации (T0→T1).
- **DoD:** публичная страница с 3D, ведущая к регистрации.

## Фаза 7 — Стабилизация и соответствие
_Связано с: NFR (152-ФЗ, безопасность, наблюдаемость)._
- [ ] Аудит-лог доступа к медданным; согласия на обработку ПДн.
- [ ] Шифрование чувствительных полей; ревизия защиты секретов MIS.
- [ ] Логи/метрики/алерты; страница состояния интеграции (FR-17).
- [ ] Прод-деплой на Beget VPS (Docker/Ubuntu) через GitHub Actions; бэкапы БД.
- **DoD:** прод-готовность, базовое соответствие 152-ФЗ.

## Зависимости (критический путь)

```
Фаза 0 ──► Фаза 1 (MIS чтение) ──► Фаза 3 (кабинеты) ──► Фаза 4 (уведомления)
   └─────► Фаза 2 (доступ) ───────┘
Фаза 1 ──► Фаза 5 (write-back)
Фаза 2 ──► Фаза 6 (лендинг/воронка)
Все ────► Фаза 7 (стабилизация)
```
Главный внешний риск — доступ и поведение API MIS (Q5a): держать Фазу 1 первой после фундамента.

## Риски и митигации

| Риск | Влияние | Митигация |
|------|---------|-----------|
| Неясны детали доступа к API MIS (Q5a) | Блокирует Фазу 1 | Запросить у MIS доступ/ApiKey и тестовый стенд до старта Фазы 1 |
| Вебхуки без гарантий доставки/подписи | Расхождение данных | Периодическая сверка + журнал `WebhookEvent` + проверка подписи если есть |
| Telegram недоступен в РФ | — (вне MVP) | В MVP только SMS; Telegram через VPS-прокси после MVP |
| Медданные / 152-ФЗ | Юридический риск | Хостинг в РФ, аудит-лог, согласия, тир-доступ (T2) |
| Гонки чтение/запись при write-back | Несогласованность | Идемпотентность, очередь повторов, сверка с вебхуком |
| Объём данных/нагрузка не определены (Q9b) | Риск для NFR | Уточнить ожидаемые объёмы; для одной клиники нагрузка низкая |
