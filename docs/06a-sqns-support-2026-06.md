## 3. Вебхуки

**3.1. Подписки `hook_settings`.**
> «Несколько URL можно — передать в массиве через запятую. `PATCH` добавляет поверх ещё URL; `POST` записывает с нуля, стирая ранее введённые».

**3.2. События.** Поддерживаемые сущности:
- `Visit` → Визит
- `Client` → Клиент
- `Service` → Услуга
- `Service` → Товар *(в ответе так; вероятно `commodity`)*
> «При смене статуса визита событие отправляется в том числе. Хук уходит, когда у нас в системе происходит событие (создание / изменение / удаление сущности)».

**3.3. Гарантии доставки.**
> «Хук уходит **единожды**. В случае ошибки или таймаута повторный хук **не уходит**».
(Ретраев нет; порядок/гарантии доставки отсутствуют.)

**3.4. Безопасность вебхуков.**
> «Подписи и общего секрета на данный момент **нет**, появится **в конце года**».
(IP-фильтрации отправителя тоже нет — см. 6.2.)

**3.5. Реальные примеры (create / update визита).**

Создание визита:
```bash
curl -X 'POST' 'https://crmexchange.1denta.ru/api/v2/visit' \
  -H 'accept: application/json' \
  -H 'authorization: Bearer *' \
  -H 'Content-Type: application/json' \
  -d '{
    "visit": {
      "user": { "name": "Fetch test logs", "phone": "+79000000000", "email": "fetch-test@arnica.pro" },
      "comment": "Fetch test for logs",
      "appointment": { "serviceIds": ["519"], "resourceId": "2", "datetime": "2026-06-09T17:00:00+05:00" }
    }
  }'
```
Пэйлоад (ответ/событие):
```json
{ "visit": { "id": 5482, "serviceIds": ["519"], "resourceId": "2", "datetime": "2026-06-09T17:00:00+05:00" } }
```

Обновление визита (можно поменять дату/время и добавить комментарий):
```bash
curl -X 'PUT' 'https://crmexchange.1denta.ru/api/v2/visit/5482' \
  -H 'accept: application/json' \
  -H 'authorization: Bearer *' \
  -H 'Content-Type: application/json' \
  -d '{ "comment": "new comment: update visit", "datetime": "2026-06-09T19:00:00+05:00" }'
```
Пэйлоад:
```json
{ "visit": { "id": [5482], "serviceIds": ["519"], "resourceId": "2", "datetime": "2026-06-09T19:00:00+05:00" } }
```
> Примечание: в примерах техподдержки `datetime` со смещением `+05:00` — это пояс их тестового аккаунта. **Наша клиника — GMT+3** (см. 4.4).

## 4. Синхронизация и формат данных

**4.1. Параметр `modificate`.**
> «Позволяет найти сущность, которая менялась за последние **N минут**. Например, `modificate=90` покажет сущности, менявшиеся за последние 90 минут. Максимальный период — **1440 минут (ровно 1 сутки)**. Больше система искать не будет».
(Это **не** timestamp и не дата — окно «минут назад», максимум 24 ч.)

**4.2. Пагинация.**
> «По содержимому `meta`. В зависимости от `perPage` меняется число страниц (`lastPage`). Актуальность и количество выгруженного проверять в `meta`».

**4.3. Лимиты запросов.**
> «**15 запросов в секунду И 900 в минуту**. При превышении отдадим **444** ошибку. Рекомендуемые интервалы зависят от активности клиники, в среднем **~15 минут**».

**4.4. Часовые пояса.**
> «Все значения с датой отдаются согласно часовому поясу клиники. Его нужно настроить в разделе „настройки → настройки клиники → Часовой пояс для рассылок“».
- **Часовой пояс нашей клиники = GMT+3** (Europe/Moscow). «Наивные» значения `datetime` без смещения трактуем как GMT+3.

**4.5. Статусы визита (`attendance`).**
- `-1` — отменён (`cancel`)
- `0` — не подтверждён (`new`)
- `1` — клиент пришёл (`showedUp`)
- `2` — подтверждён (`confirmed`)

Соответствуют значениям `PUT /api/v2/visit/{id}/status` = `new / confirmed / showedUp / cancel`.
> Переходы: «При наличии распечатанного чека статус менять нельзя — зафиксирован `showedUp`. В любом другом случае можно менять как угодно и сколько угодно, **пока не наступило время начала визита**».

**4.6. Врачи и ресурсы.**
> «Да, id `resource` и `employee` совпадают» (`resource` адресуется параметром `employeeId`; ресурс = сотрудник).

## 5. Создание и изменение записей (write-back)

**5.1. `POST /api/v2/visit` — обязательные поля и матчинг клиента.**
> «Если клиент с таким номером есть — к нему подтянутся данные из запроса (если ФИО новое, в карте пациента оно **перезапишется**). Если клиента с таким телефоном нет — создаётся новый».

Пример тела:
```json
{
  "visit": {
    "user": { "name": "Fetch test logs", "phone": "+79000000000", "email": "fetch-test@arnica.pro" },
    "comment": "Fetch test for logs",
    "appointment": { "serviceIds": ["519"], "resourceId": "2", "datetime": "2026-05-20T13:00:00+05:00" }
  }
}
```

**5.2. Отмена записи — `DELETE` vs `PUT .../status=cancel`.**
> «`DELETE` мягко удаляет визит (в справочнике визитов его всё равно можно найти) и освобождает слот. `PUT` просто отменяет визит, но также освобождает слот. Если надо, чтобы визит отображался в статистике — лучше отменять через `PUT`».

**5.3. Свободные слоты.**
> «Обязательные поля — через swagger. Длительность слота считается по „Интервалу записи“ в настройках онлайн-записи. Если интервал 30 мин — слоты по API будут 30 мин. В карточке сотрудника есть индивидуальный интервал — при наличии он приоритетнее. Занятые слоты учитываются при ответе».

**5.4. Идемпотентность создания.**
> «Если в программе слот занят, а в вашей свободен, то при попытке занять его `POST /api/v2/visit` мы отдадим ошибку `"code": "SlotUnavailableError"`».
(Отдельного ключа идемпотентности нет — защита от дублей на нашей стороне.)
